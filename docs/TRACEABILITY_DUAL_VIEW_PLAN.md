# План внедрения второго вида трассировки (две направленные таблицы)

## 1. Контекст и текущая реализация

На текущий момент в GUI трассировка реализована через матрицу (`TraceMatrixFrame`) и диалог конфигурации (`TraceMatrixConfigDialog`) в `app/ui/trace_matrix.py`. Пользователь уже может выбирать:

- документы для строк/столбцов;
- направление (`child→parent` / `parent→child`);
- поля карточки требований для подписи/экспорта;
- сортировку строк/столбцов по одному полю;
- формат вывода (`interactive`, `matrix-html`, `matrix-csv`, `matrix-json`).

Данные матрицы формируются в `app/core/trace_matrix.py` и вызываются через `DocumentsController.build_trace_matrix()`.

### Выявленные расхождения с новой постановкой

Требуется дополнительно к матрице реализовать:

1. **Все режимы отображения** на форме генерации:
   - матрица;
   - две направленные таблицы;
   - одновременно матрица + две таблицы.
2. **Раздельный выбор полей по уровням**:
   - поля верхнего уровня;
   - поля нижнего уровня.
3. **Экспорт нового результата** в `HTML`, `PDF`, `MD`.
4. **Минимальные настройки дизайна отрисовки**:
   - цвета;
   - шрифт;
   - размер;
   - отображение стрелок.
5. **При отсутствии связи показывать прочерк**.
6. **Полный функционал сортировки и фильтрации по каждому столбцу** (как на главной форме списка требований).
7. **Большую продуманную форму настроек** именно для генерации трассировки.

## 2. Целевая архитектура

## 2.1. Разделение доменной модели и представлений

Расширить `app/core/trace_matrix.py` (или выделить рядом модуль `app/core/trace_views.py`) так, чтобы один доменный слой формировал сразу несколько представлений из единого набора связей:

- `TraceMatrixViewData` (текущая матрица);
- `TraceForwardTableViewData` (верхний → нижний);
- `TraceReverseTableViewData` (нижний → верхний).

Ключевой принцип: GUI рендерит готовые view-model и не содержит вычислений трассировочной семантики.

## 2.2. Единый конфиг генерации

Ввести агрегирующий конфиг (рабочее имя `TraceViewPlanConfig`) с секциями:

- `base`:
  - документы верхнего/нижнего уровня;
  - направление базовой матрицы (если применимо);
- `modes`:
  - `show_matrix: bool`;
  - `show_directional_tables: bool`;
- `fields`:
  - `top_fields: tuple[str, ...]`;
  - `bottom_fields: tuple[str, ...]`;
  - дефолт: `("rid", "title")` для обоих уровней;
- `grid_behavior`:
  - сортировка по колонкам;
  - фильтры по колонкам (на уровне каждой таблицы);
- `render_style`:
  - цвета (палитра строк/ячеек/акцентов);
  - шрифт (family);
  - размер шрифта;
  - включение/стиль стрелок;
- `export`:
  - формат: `interactive | html | pdf | md`;
  - целевой путь.

Хранение/восстановление настроек — через `wx.Config` аналогично текущей схеме (`trace_matrix/dialog`, `trace_matrix/frame`), но с новым префиксом (например, `traceability_view/*`) для управляемой миграции.

## 2.3. Модель строк для таблиц

Для обеих новых таблиц использовать единый формат строки:

- `source_requirement` (требование текущего уровня);
- `source_display` (значения выбранных полей уровня);
- `linked_requirements` (список требований противоположного уровня);
- `linked_display` (подготовленные строковые представления);
- `has_links: bool`.

Если `has_links == False`, в целевом поле/ячейке отображается `"—"`.

## 3. План реализации по шагам

## Шаг 1. Домен и контроллеры

1. Добавить новый API в контроллер документов (`app/ui/controllers/documents.py`):
   - `build_trace_views(config) -> TraceViewsBundle`.
2. В доменном слое реализовать построение двунаправленных таблиц из тех же данных, что питают матрицу.
3. Добавить тесты ядра на:
   - корректность прямого и обратного отображения;
   - поведение при отсутствии связей (прочерк);
   - корректность раздельных наборов полей.

## Шаг 2. Новая большая форма конфигурации

1. Расширить/заменить `TraceMatrixConfigDialog` на «большую» форму (`TraceabilityViewConfigDialog`) с блоками:
   - выбор документов уровней;
   - режимы отображения (все режимы);
   - поля верхнего уровня;
   - поля нижнего уровня;
   - сортировка/фильтры;
   - дизайн;
   - экспорт.
2. Реализовать полноценную валидацию:
   - хотя бы один режим выбран;
   - хотя бы одно поле на каждый уровень (иначе авто-дефолт `rid,title`);
   - проверка допустимости экспортного формата.
3. Реализовать persist/restore настроек (включая размеры окна, наборы полей, фильтры и стиль).

## Шаг 3. UI двух таблиц с «полным фаршем» сортировки/фильтрации

1. Реализовать отдельный компонент представления направленной таблицы (рабочее имя `TraceDirectionalTablePanel`).
2. Взять за основу поведение главного списка (`app/ui/list_panel.py`):
   - сортировка кликом по каждому столбцу;
   - фильтрация (query + field filters + status/labels при применимости);
   - сброс фильтров;
   - отображение summary по активным фильтрам.
3. Сделать два экземпляра панели:
   - верхний → нижний;
   - нижний → верхний.
4. Для несвязанных строк в целевой колонке показывать `—`.

## Шаг 4. Рендер режимов

1. В точке вызова из `MainFrame.on_show_trace_matrix` (файл `app/ui/main_frame/documents.py`) внедрить маршрутизацию по режимам:
   - открыть окно матрицы;
   - открыть окно двух таблиц;
   - открыть комбинированный режим (вкладки или сплит-вид).
2. Для комбинированного режима выбрать безопасный UX:
   - `wx.Notebook` с вкладками `Matrix / Top→Bottom / Bottom→Top` (предпочтительно для производительности и читаемости).

## Шаг 5. Экспорт HTML/PDF/MD

1. Ввести новый слой экспортёров трассировки:
   - `trace-html` (обновить текущий html-рендер, чтобы поддерживал таблицы);
   - `trace-md` (табличный markdown + секции для двух направлений);
   - `trace-pdf`.
2. По PDF:
   - использовать уже применяемый в проекте путь генерации PDF (если есть готовый pipeline);
   - если в проекте нет унифицированного рендера PDF, реализовать через HTML→PDF с контролируемым fallback и детерминированными стилями.
3. Экспорт должен поддерживать:
   - матрицу;
   - две таблицы;
   - комбинированный режим.

## Шаг 6. Настройки дизайна (минимальный набор)

1. Добавить UI для минимальных стилей:
   - основной цвет акцентов;
   - цвет linked/unlinked;

2. Применить настройки:

   - в HTML/MD/PDF экспорте (где применимо).
3. Добавить безопасные дефолты и кнопку «Сбросить стиль».

## Шаг 7. Локализация и документация

1. Обновить строки локализации в:

## 5. Риски и меры

1. **Производительность без пагинации** (прямое требование: показывать весь объём).
   - Меры: виртуальные таблицы (`GridTableBase`/lazy access), кеш форматированных ячеек, ограничение дорогостоящих пересчётов фильтров.
2. **Сложность UX из-за «полного фарша» фильтров/сортировок.**
   - Меры: унифицировать поведение с `ListPanel`, переиспользовать знакомые паттерны и текстовые подсказки.
3. **PDF-экспорт в headless среде.**
   - Меры: выбрать deterministic backend; заранее добавить smoke-тест на CI-среду.
4. **Риск дублирования кода между матрицей и таблицами.**
   - Меры: общий `TraceViewsBundle` и общие форматтеры полей.
5. **Высокая вероятность пропусков в локализации.**
   - Меры: обязательный чек обновления `en/ru` `.po` в Definition of Done.

## 7. Найденные технические долги/улучшения по пути анализа

1. В текущей реализации трассировки есть разделение «конфиг диалога» и «логика отображения», но часть поведения (например, fallback для полей, форматы output) распределена между UI и экспортными хелперами; стоит централизовать в единый config schema.
2. Формат экспорта сейчас ориентирован на `matrix-*`; имеет смысл заранее унифицировать naming (`trace-html`, `trace-md`, `trace-pdf`, возможно `trace-json`) и предусмотреть backward compatibility для CLI.
3. Для «полного фарша» фильтрации стоит заранее выделить повторно используемый FilterState/FilterEngine, чтобы не копировать логику `ListPanel` в trace-панели.
