# План стабилизации истории агента (правый чат-панель)

## Контекст проблемы

Симптом из пользовательских сценариев: элементы истории хода агента (reasoning, текстовые ответы, вызовы tool) иногда отображаются не в реальной последовательности исполнения — блоки могут «съезжать» и группироваться по типу вместо хронологии шага.

По текущему коду проблема действительно структурная:

- История собирается из нескольких источников одновременно (`payload.timeline`, `events`, `tool_results`, `llm_trace`, `diagnostic`, поля `entry.response*`).
- В `view_model` и `history_utils` есть несколько fallback-путей, которые в разных состояниях (стриминг, завершённый запуск, чтение старой истории) дают разный порядок.
- Есть разделение между "authoritative timeline" и "missing/damaged", из-за чего рендерер иногда перестраивает порядок по timestamp/step и меняет раскладку между перерисовками.

## Цель

Сделать **один канонический путь** построения и отображения истории хода агента:

1. один формат событий,
2. один алгоритм сортировки/нумерации,
3. одна точка нормализации для live + completed + loaded-from-file,
4. предсказуемая деградация только при реально битых данных,
5. полный тестовый контур (unit + GUI smoke + интеграционные сценарии).

## Предлагаемая архитектура целевого состояния

## 1) Ввести единый «канонический event stream»

Вместо текущего смешивания источников в UI ввести слой `AgentCanonicalTimelineService` (рабочее имя):

- вход: `AgentRunPayload` + минимально `ChatEntry` (timestamp и идентификатор),
- выход: `tuple[AgentTimelineEntry]` + metadata (`checksum`, `status`, diagnostics).

Правила:

- Если в payload есть валидный `timeline` + checksum ок → используется как source of truth.
- Если `timeline` отсутствует/битый → **однократно** восстанавливаем из `events + llm_trace + tool_results` через `build_agent_timeline`, сохраняем как canonical и checksum.
- После восстановления всегда работаем только с canonical timeline (никаких параллельных повторных merge в рендере).

## 2) Убрать двойную сортировку в UI

Сейчас `_build_agent_turn()` сортирует финальные `events` вторично (`_event_sort_key`), что может конфликтовать с уже заданной sequence timeline.

Цель:

- если есть canonical timeline → порядок **только по sequence**,
- timestamp/step используются только для отображения, не для reorder,
- fallback сортировка по timestamp допустима только если timeline отсутствует полностью.

## 3) Нормализовать жизненный цикл данных

Единый pipeline в 3 состояниях:

1. **Live-run update**: при каждом новом шаге обновляется canonical payload (с checksum) и UI получает уже нормализованный snapshot.
2. **Completed run**: финализация не перестраивает порядок заново, только проставляет недостающие status/timestamps.
3. **History load**: при чтении истории выполняется canonicalization-on-read (лениво + кеш), затем данные сохраняются в памяти в каноническом формате.

## 4) Явная политика деградации

Вместо множества скрытых fallback:

- `timeline_status=valid` → строгий порядок sequence.
- `timeline_status=recovered` (новый) → восстановлено автоматически, но порядок фиксирован.
- `timeline_status=damaged` → показываем диагностическое системное сообщение, но не пытаемся «умно» перемешивать блоки по типам.

## Пошаговый план реализации

## Этап A. Диагностика и фиксация инвариантов

1. Задокументировать инварианты timeline в отдельном модуле/доке:
   - sequence непрерывный: 0..N-1,
   - события одного шага идут в порядке генерации,
   - `tool_call` никогда не переезжает относительно породившего его `llm_step`.
2. Добавить debug-лог «timeline decision trace» (какой источник выбран и почему).
3. Зафиксировать проблемный сценарий пользователя как regression fixture (похоже на пагинацию `list_requirements` по 50 элементов).

## Этап B. Канонизатор

1. Вынести построение canonical timeline из UI (`view_model`) в отдельный сервис в `app/agent` или `app/ui/agent_chat_panel/history_*`.
2. Перевести `ensure_canonical_agent_payload()` в роль единственной точки подготовки payload перед:
   - сохранением,
   - рендерингом,
   - экспортом transcript.
3. Удалить дублирующие ветки, где timeline вычисляется по месту в UI.

## Этап C. Рендеринг без ре-оркестрации

1. В `_build_agent_events()` принимать уже упорядоченный canonical stream.
2. Убрать/свести к минимуму `_event_sort_key`-логику при наличии timeline.
3. Проверить, что partial updates в live режиме не пересобирают прошлые sequence.

## Этап D. Миграция старых историй

1. При загрузке старых payload без timeline делать мягкую миграцию в памяти.
2. При следующем сохранении записывать canonical timeline + checksum.
3. Добавить версию формата истории (например, `timeline_schema_version`) для прозрачной эволюции.

## Этап E. Чистка fallback-веток

1. Провести ревизию fallback-логики в:
   - `view_model.py`,
   - `history_utils.py`,
   - `history_store.py`,
   - `log_export.py`.
2. Удалить неиспользуемые/дублирующие ветви вместо «оставить на всякий случай».
3. Обновить архитектурную документацию и dev-notes о едином потоке.

## Стратегия тестирования ("обложить всё")

## 1) Unit: канонизация и порядок

Новые/расширенные тест-группы:

- `test_agent_timeline_canonicalization.py`
  - timeline из payload валиден → порядок не меняется,
  - timeline отсутствует → корректное восстановление,
  - timeline битый checksum → recover + статус `recovered`.
- `test_agent_chat_event_order.py`
  - interleaving: reasoning → text → tool → reasoning → tool,
  - одинаковые timestamps у разных типов событий,
  - пропуски step_index.
- `test_agent_chat_live_updates.py`
  - последовательные partial snapshots не ломают уже выданный sequence,
  - новый шаг добавляется только в хвост.

## 2) Unit: совместимость истории

- загрузка legacy payload без `timeline`,
- загрузка payload только с `tool_results`,
- смешанные кривые поля (`diagnostic.llm_steps`, частично пустые timestamps),
- проверка, что итоговый conversation export использует канонический порядок.

## 3) GUI smoke (обязательно при этой задаче)

Фокус:

- `tests/gui/test_agent_chat_panel.py` + отдельный regression case с зафиксированным expected-order,
- сценарий «долгий многошаговый run с несколькими tool calls» под `pytest --suite gui-smoke -q`.

Проверки:

- порядок виджетов в правой панели соответствует sequence,
- после очередного live-update ранее отрисованные шаги не "перетасовываются",
- после перезагрузки истории порядок идентичен live-версии.

## 4) Integration

- расширить `tests/integration/test_llm_openrouter_transcript_logs.py`/смежные,
- проверить полный цикл: run → persist → reload → export.

Критерий: checksum одинаков до и после reload, порядок событий идентичен.

## 5) Нагрузочный/стабильность

- синтетический сценарий 100+ timeline events,
- несколько последовательных regenerate в одном диалоге,
- repeated open/close окна без дрейфа порядка.

---

## Критерии готовности (Definition of Done)

1. Для всех режимов (live/completed/reload) порядок единообразный и следует sequence.
2. Нет визуальной перегруппировки блоков при очередном update.
3. Legacy истории без timeline открываются без падений и мигрируются в canonical вид.
4. Добавлены regression-тесты на сценарий с пагинацией и многошаговыми tool calls.
5. Документация архитектуры обновлена под новый единый pipeline.

## Риски и как их закрывать

- **Риск:** сломать старые экспорты/диагностику, где ожидался старый fallback.
  - **Митигировать:** golden tests на log export до/после.
- **Риск:** живой стриминг может приходить неупорядоченно по времени.
  - **Митигировать:** sequence-first политика + append-only контракт.
- **Риск:** часть тестов опирается на старую нестабильную сортировку.
  - **Митигировать:** массовое обновление ожиданий + фиксация новых инвариантов.
